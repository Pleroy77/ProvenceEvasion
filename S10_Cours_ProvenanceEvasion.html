1. Une faille de sécurité identifiée depuis l'audit

📍 Mise en situation : Depuis S1, vous administrez tous les équipements de Provence Évasion (switches, routeurs) en local, sur le réseau de l'établissement. Mais l'audit initial d'Axentis Informatique pointait déjà un manque : « Aucun accès distant sécurisé — Laurent Ferracci ne peut pas consulter les données de ses sites depuis l'extérieur. » Et il y a un problème plus immédiat encore : jusqu'à présent, vous vous êtes connectés aux équipements via Telnet, un protocole vieux de plus de 40 ans qui ne chiffre rien. Aujourd'hui, on corrige les deux problèmes : sécuriser l'administration locale (SSH) et préparer l'accès distant sécurisé (VPN).

1.1 Telnet : un protocole non sécurisé

Telnet (port TCP 23) permet d'ouvrir une session d'administration à distance sur un équipement réseau. Il a été conçu dans les années 1970, à une époque où la sécurité n'était pas une priorité : toutes les données transitent en clair, y compris les identifiants et mots de passe.

⚠ Danger concret : Si un attaquant a accès à un point quelconque du chemin réseau entre le poste d'administration et l'équipement (un commutateur compromis, un point d'accès WiFi mal sécurisé, un câble sur lequel on branche un répéteur réseau...), il peut utiliser un analyseur de trames (sniffer) comme Wireshark et lire directement le mot de passe administrateur dans les paquets capturés. Une fois ce mot de passe obtenu, il peut prendre le contrôle total de l'équipement.

2. SSH — Secure Shell

2.1 Principe de fonctionnement

SSH (port TCP 22) remplace Telnet en apportant ce qui lui manque totalement : la sécurité. À l'ouverture d'une session SSH, le client et l'équipement négocient d'abord un échange de clés (chiffrement asymétrique) qui permet de générer une clé de session partagée, sans jamais la faire transiter en clair sur le réseau. Toute la suite de la session est ensuite chiffrée avec cette clé (chiffrement symétrique, plus rapide). SSH garantit ainsi trois choses : la confidentialité (personne ne peut lire les échanges), l'intégrité (les données ne peuvent pas être modifiées sans être détectées) et l'authentification (on peut vérifier l'identité de l'équipement et de l'utilisateur).

📌 SSH vs Telnet

Telnet : port 23, aucun chiffrement, données en clair, à proscrire sur un équipement de production.
SSH : port 22, chiffrement des échanges (clés asymétriques puis symétriques), authentification renforcée, standard actuel pour toute administration à distance.

2.2 Pré-requis à la configuration SSH sur un équipement Cisco

Avant d'activer SSH, l'équipement a besoin de générer sa propre paire de clés RSA. Pour cela, IOS exige trois éléments préalables : un nom d'hôte unique, un nom de domaine, et une base d'utilisateurs locaux pour l'authentification.

# Pré-requis et activation de SSH sur un routeur Cisco Router> enable Router# configure terminal Router(config)# hostname R-Apt R-Apt(config)# ip domain-name provence-evasion.local R-Apt(config)# crypto key generate rsa % Choose a key modulus between 360 and 2048 How many bits in the modulus [512]: 1024 R-Apt(config)# username admin secret MotDePasse_Solide! R-Apt(config)# ip ssh version 2 R-Apt(config)# line vty 0 4 R-Apt(config-line)# transport input ssh R-Apt(config-line)# login local R-Apt(config-line)# exec-timeout 5 0

3. Le VPN — chiffrer les communications sur un réseau non maîtrisé

3.1 Pourquoi un VPN ?

SSH sécurise l'administration locale, mais ne résout pas tout. Deux besoins identifiés chez Provence Évasion impliquent de faire transiter des données en dehors du réseau local de l'entreprise, potentiellement via Internet : la liaison entre les sites (Apt, Roussillon, Saignon, Cucuron) si elle passe par un opérateur public, et l'accès de Laurent Ferracci à ses données depuis l'extérieur. Un VPN (Virtual Private Network) crée un tunnel chiffré entre deux points, garantissant que les données restent confidentielles même si elles traversent un réseau non maîtrisé.

3.2 Deux grandes familles de VPN

📌 VPN site-à-site

Relie de façon permanente deux réseaux d'entreprise à travers un équipement de passerelle à chaque extrémité (typiquement deux routeurs). Aucune action n'est requise côté utilisateur final : la communication est automatiquement chiffrée dès qu'elle traverse le tunnel. C'est la solution adaptée pour relier en continu le réseau d'Apt à celui de Roussillon, par exemple.

📌 VPN d'accès distant (client-to-site)

Permet à un utilisateur isolé (un poste, un smartphone) de se connecter temporairement au réseau de l'entreprise via un logiciel client VPN. C'est la solution adaptée pour Laurent Ferracci, qui souhaite consulter les données du siège depuis son domicile de façon occasionnelle.

⚠ Limite pédagogique : Cisco Packet Tracer permet de configurer un véritable tunnel VPN site-à-site IPsec en CLI sur les routeurs, mais ne simule pas correctement un client VPN d'accès distant (type Cisco AnyConnect). C'est pourquoi, dans l'atelier de cette séance, vous configurerez uniquement le VPN site-à-site entre Apt et Roussillon. Le cas d'usage de Laurent Ferracci reste traité ici au niveau conceptuel : vous devez en comprendre le principe, sans le configurer dans Packet Tracer.

4. Le VPN IPsec site-à-site — les deux phases de négociation

La mise en place d'un tunnel VPN IPsec entre deux routeurs Cisco se déroule en deux phases successives, normalisées par le protocole IKE (Internet Key Exchange), qui s'appuie sur ISAKMP.

4.1 Phase 1 — ISAKMP/IKE

La phase 1 établit un canal sécurisé entre les deux routeurs, avant même de chiffrer les données utilisateur. Les deux équipements négocient une politique de sécurité commune (algorithme de chiffrement, fonction de hachage, groupe Diffie-Hellman, durée de vie) puis s'authentifient mutuellement, le plus souvent via une clé pré-partagée (pre-shared key) identique configurée sur les deux routeurs.

4.2 Phase 2 — IPsec

Une fois le canal sécurisé établi, la phase 2 négocie les paramètres de chiffrement des données utilisateur elles-mêmes, via un transform-set (algorithme de chiffrement + algorithme d'intégrité). Trois éléments supplémentaires sont nécessaires :

  • Une ACL de « trafic intéressant » : elle définit précisément quel trafic doit être chiffré dans le tunnel (par exemple : du réseau d'Apt vers le réseau de Roussillon, et inversement). Tout le reste du trafic continue de circuler normalement, sans chiffrement.
  • Une crypto map : elle assemble l'ACL de trafic intéressant, l'adresse du routeur distant (peer) et le transform-set en une seule configuration.
  • L'application de la crypto map sur l'interface WAN du routeur (celle qui fait face au réseau non maîtrisé), pour que le chiffrement s'applique au bon trafic sortant.
Schéma — Tunnel VPN IPsec site-à-site Apt ↔ Roussillon
[ LAN Apt ] [ LAN Roussillon ] │ │ [ R-Apt ] ── interface WAN ══════ Tunnel IPsec ══════ interface WAN ── [ R-Roussillon ] (chiffré : trafic LAN Apt ↔ LAN Roussillon "intéressant" selon l'ACL — Phase 1 ISAKMP puis Phase 2 IPsec, crypto map appliquée sur chaque interface WAN)
