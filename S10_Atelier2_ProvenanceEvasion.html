1. Contexte — pourquoi chiffrer aussi la liaison inter-sites

📍 Mise en situation : En Atelier 1, vous avez sécurisé l'administration locale des 8 équipements grâce à SSH. Mais la communication entre les VLANs d'Administration d'Apt et de Roussillon transite par la liaison WAN inter-sites établie en S5-S6 — une liaison qui, dans un déploiement réel, traverserait Internet via les box des opérateurs. Aujourd'hui, vous allez chiffrer cette liaison grâce à un tunnel VPN IPsec site-à-site entre les routeurs R-Apt et R-Roussillon. En fin de séance, vous compléterez la Section 11 du dossier de déploiement.
🎯 Objectifs de l'Atelier 2
  • Identifier les interfaces WAN de R-Apt et R-Roussillon utilisées comme extrémités du tunnel
  • Configurer la Phase 1 ISAKMP/IKE (politique de sécurité + clé pré-partagée) sur les deux routeurs
  • Configurer la Phase 2 IPsec (transform-set, ACL de trafic intéressant, crypto map) sur les deux routeurs
  • Appliquer la crypto map sur l'interface WAN de chaque routeur
  • Vérifier l'établissement du tunnel et tester la connectivité chiffrée
  • Diagnostiquer une panne de tunnel VPN
  • Rédiger la Section 11 du dossier de déploiement
📋 Consigne : Ouvrez votre fichier PE_Complet_S10_Atelier1.pkt (créé en Atelier 1). Sauvegardez-le immédiatement sous le nouveau nom PE_Complet_S10_Atelier2.pkt avant toute modification.

3. Phase 1 — Configuration ISAKMP/IKE

La phase 1 doit être configurée de façon strictement symétrique sur les deux routeurs : mêmes paramètres de chiffrement, de hachage et de groupe Diffie-Hellman, et même clé pré-partagée.

# Phase 1 ISAKMP/IKE — squelette de configuration (à adapter) Router(config)# crypto isakmp policy 10 Router(config-isakmp)# encryption aes Router(config-isakmp)# hash sha Router(config-isakmp)# authentication pre-share Router(config-isakmp)# group 2 Router(config-isakmp)# lifetime 86400 Router(config-isakmp)# exit Router(config)# crypto isakmp key ProvenceVPN2026! address <adresse WAN du routeur distant>
📝 Exercice — Phase 1 sur R-Apt et R-Roussillon

4. Phase 2 — IPsec : transform-set, ACL et crypto map

La phase 2 définit comment chiffrer les données utilisateur, et surtout quel trafic doit transiter dans le tunnel. Pour cet atelier, seul le trafic entre les VLANs d'Administration d'Apt (192.168.14.0/28) et de Roussillon (192.168.24.0/28) sera considéré comme « intéressant » et donc chiffré.

# Phase 2 IPsec — squelette de configuration sur R-Apt R-Apt(config)# access-list 110 permit ip 192.168.14.0 0.0.0.15 192.168.24.0 0.0.0.15 R-Apt(config)# crypto ipsec transform-set TS-PE esp-aes esp-sha-hmac R-Apt(config)# crypto map CMAP-PE 10 ipsec-isakmp R-Apt(config-crypto-map)# set peer <adresse WAN R-Roussillon> R-Apt(config-crypto-map)# set transform-set TS-PE R-Apt(config-crypto-map)# match address 110 R-Apt(config-crypto-map)# exit R-Apt(config)# interface <interface WAN> R-Apt(config-if)# crypto map CMAP-PE

5. Vérification et tests

Étape 1 Depuis un PC du VLAN Administration d'Apt, envoyez un ping vers l'adresse de gestion d'un équipement du VLAN Administration de Roussillon (192.168.24.2). Ce trafic correspond à l'ACL de trafic intéressant et doit déclencher la négociation du tunnel.
Étape 2 Sur R-Apt, exécutez successivement show crypto isakmp sa puis show crypto ipsec sa.

6. Scénario de dépannage

🛠 Diagnostic d'une panne de tunnel VPN

Cette partie vous demande de réfléchir comme un technicien face à un dysfonctionnement, sans avoir la configuration complète sous les yeux.

🛠 Scénario — Tunnel qui ne s'établit pas

Un collègue a configuré le tunnel VPN entre Apt et Roussillon, mais le ping entre les deux VLANs Administration échoue toujours. Voici ce qu'il vous transmet :

  • Sur R-Apt, crypto isakmp key ProvenceVPN2026! address <WAN Roussillon> a bien été saisi.
  • Sur R-Roussillon, crypto isakmp key ProvenceVPN2026 address <WAN Apt> a été saisi (sans le point d'exclamation final).
  • show crypto isakmp sa ne montre aucune SA active sur les deux routeurs.
