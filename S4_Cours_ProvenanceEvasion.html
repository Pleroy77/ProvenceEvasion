1. Pourquoi le DHCP ? — Rappels et mise en situation

1.1 Le problème de l'adressage manuel

🏨 Situation Provence Évasion : Le groupe dispose de 4 sites (Apt, Roussillon, Saignon, Cucuron). Rien que sur le site d'Apt (le siège), on compte une cinquantaine de postes : PCs du personnel, téléphones IP, imprimantes réseau, tablettes WiFi, caméras IP… Si un technicien devait configurer manuellement l'adresse IP de chaque équipement, il passerait des heures sur chaque site, avec un risque élevé d'erreur ou de doublon d'adresse.

Sans DHCP, chaque équipement réseau doit être configuré à la main avec :

  • Une adresse IP (unique dans le réseau)
  • Un masque de sous-réseau
  • Une passerelle par défaut
  • L'adresse du serveur DNS

Cette méthode, appelée adressage statique, est acceptable pour les équipements fixes importants (serveurs, routeurs, switches, caméras fixes). Mais elle est ingérable à grande échelle pour les postes utilisateurs, les équipements mobiles ou les invités WiFi.

1.2 La solution : le protocole DHCP

DHCP signifie Dynamic Host Configuration Protocol (protocole de configuration dynamique des hôtes). Il a été standardisé par le RFC 2131 en 1997.

Son rôle est simple : quand un équipement se connecte au réseau, il demande automatiquement une configuration IP à un serveur DHCP. Ce serveur lui attribue une adresse disponible, ainsi que toutes les informations nécessaires.

📌 À retenir — Ce que le DHCP fournit automatiquement
Un serveur DHCP distribue à chaque client qui en fait la demande :

  • Une adresse IP disponible dans la plage définie
  • Le masque de sous-réseau
  • L'adresse de la passerelle par défaut (Default Gateway)
  • L'adresse du ou des serveurs DNS
  • La durée de bail (Lease Time) : combien de temps l'adresse est réservée pour cet équipement

2. Le protocole DORA — Comment le DHCP fonctionne

2.1 Les 4 étapes du protocole

Quand un équipement se connecte au réseau sans adresse IP configurée, il déclenche automatiquement un échange en 4 étapes, résumé par l'acronyme DORA :

D
DISCOVER
Le client envoie un message en broadcast (diffusion générale) pour trouver un serveur DHCP disponible.

Adresse source : 0.0.0.0
Adresse dest. : 255.255.255.255
O
OFFER
Le serveur DHCP répond en proposant une adresse IP disponible ainsi que le masque, la passerelle et la durée de bail.

La proposition est encore en broadcast
R
REQUEST
Le client accepte l'offre et envoie une demande officielle d'utilisation de l'adresse proposée (toujours en broadcast, pour prévenir les autres serveurs).

Il peut y avoir plusieurs serveurs DHCP
A
ACKNOWLEDGE
Le serveur confirme l'attribution. Le client peut désormais utiliser l'adresse IP. Le bail démarre.

Le serveur enregistre le bail dans sa table

2.2 Schéma de l'échange DORA

Échange DORA — Client DHCP ↔ Serveur DHCP (site d'Apt)
  CLIENT (PC-Accueil-1)                    SERVEUR DHCP (Apt)
  [Pas encore d'adresse IP]                [192.168.11.1 — VLAN Administration]

         │                                         │
         │──── DISCOVER ──────────────────────────►│
         │     src: 0.0.0.0  dst: 255.255.255.255  │
         │     "Y a-t-il un serveur DHCP ?"        │
         │                                         │
         │◄─── OFFER ──────────────────────────────│
         │     "Je te propose 192.168.12.10"       │
         │     Masque / GW / DNS / Bail : 8h       │
         │                                         │
         │──── REQUEST ───────────────────────────►│
         │     "J'accepte l'adresse 192.168.12.10" │
         │     (broadcast pour annuler autres srv) │
         │                                         │
         │◄─── ACKNOWLEDGE ────────────────────────│
         │     "C'est confirmé, bail accordé."     │
         │     Le client peut l'utiliser.          │
         │                                         │

2.3 La notion de bail (Lease Time)

Une adresse DHCP n'est pas attribuée définitivement. Elle est accordée pour une durée limitée appelée bail (lease en anglais).

  • Quand 50 % du bail est écoulé, le client tente de le renouveler auprès du serveur.
  • Quand 87,5 % du bail est écoulé, le client envoie un Discover en broadcast pour trouver n'importe quel serveur.
  • Si le bail expire sans renouvellement, l'adresse est libérée et redevient disponible.
📌 À retenir — Durée de bail recommandée selon l'usage
  • Courte (1h à 4h) : VLAN clients WiFi (nombreux appareils éphémères, forte rotation)
  • Moyenne (8h) : VLAN personnel (journée de travail, poste souvent le même)
  • Longue (24h à 7j) : équipements quasi-fixes (imprimantes réseau par exemple)
  • Très longue ou statique : serveurs, routeurs, switches, caméras

3. Serveur DHCP dédié vs DHCP sur routeur

3.1 Deux façons de déployer le DHCP

🖥 Serveur DHCP dédié
Un équipement (PC, NAS, machine virtuelle…) est configuré spécifiquement pour faire office de serveur DHCP sur le réseau.

Avantages :
 • Performances et fiabilité élevées
 • Gestion centralisée de nombreux baux
 • Journaux détaillés, statistiques
 • Peut gérer plusieurs VLANs

Inconvénients :
 • Nécessite un équipement supplémentaire
 • Plus complexe à configurer au départ
🔌 DHCP sur routeur Cisco IOS
Le routeur Cisco intègre une fonction DHCP native. Aucun serveur supplémentaire n'est nécessaire.

Avantages :
 • Pas d'équipement supplémentaire
 • Simple à déployer sur petits réseaux
 • Intégré à la configuration IOS

Inconvénients :
 • Limité en nombre de baux
 • Charge supplémentaire sur le routeur
 • Journaux moins détaillés
🏨 Choix pour Provence Évasion : Dans le cadre de ce projet, nous utiliserons la fonction DHCP intégrée aux routeurs Cisco IOS (émulés dans Packet Tracer). Dans une infrastructure réelle de la taille de Provence Évasion, Axentis préconiserait un serveur DHCP dédié (type Windows Server DHCP ou ISC DHCP sous Linux) pour plus de fiabilité.

3.2 Le problème du broadcast et les VLANs

Rappel important : un message DHCP Discover est un broadcast. Or, les broadcasts sont bloqués aux frontières des VLANs et des sous-réseaux. Si un client dans le VLAN 12 (Accueil Apt) envoie un Discover, ce message n'atteindra pas naturellement un serveur DHCP situé dans le VLAN 11 (Administration Apt).

La solution : le DHCP Relay Agent (aussi appelé ip helper-address sur Cisco). On configure le routeur pour qu'il relaye les broadcasts DHCP vers l'adresse unicast du serveur DHCP centralisé.

DHCP Relay — Fonctionnement multi-VLAN (site d'Apt)
  VLAN 12 — Accueil                    ROUTEUR APT                  VLAN 11 — Admin
  [PC-Accueil-1]                       [interface Gi0/0.12]         [Serveur DHCP]
  [0.0.0.0]                            ip helper-address            [192.168.11.5]
       │                               192.168.11.5                        │
       │──── DISCOVER (broadcast) ───►│                                    │
       │                              │──── REQUEST (unicast) ────────────►│
       │                              │◄─── OFFER (unicast) ───────────────│
       │◄─── OFFER ───────────────────│                                    │

Dans Packet Tracer, comme nous configurons le DHCP directement sur le routeur, le relay n'est pas nécessaire : le routeur est à la fois le relay et le serveur.

4. Configuration DHCP sur Cisco IOS — Commandes complètes

4.1 Structure du plan d'adressage d'Apt (rappel)

Pour configurer correctement le DHCP, il faut connaître parfaitement le plan d'adressage du site. Voici un rappel des VLANs du Site d'Apt (siège social) :

VLANNomRéseauMasquePasserelle (.1)Plage DHCPBail
11 Administration 192.168.11.0 /28 192.168.11.1 192.168.11.10 – .14 8h
12 Personnel Accueil 192.168.12.0 /27 192.168.12.1 192.168.12.10 – .30 8h
13 Clients WiFi 192.168.13.0 /25 192.168.13.1 192.168.13.10 – .126 2h
14 Caméras IP 192.168.14.0 /28 192.168.14.1 — Statique
15 Serveurs 192.168.15.0 /28 192.168.15.1 — Statique
⚠ Important : Les caméras IP et les serveurs ont des adresses statiques. On ne configure pas de pool DHCP pour ces VLANs. Les adresses de la passerelle (.1) et des équipements fixes sont également exclues de la distribution DHCP pour éviter tout conflit.

4.2 Étape 1 — Exclure les adresses réservées

Avant de définir un pool DHCP, on commence toujours par exclure les adresses qui ne doivent pas être distribuées automatiquement : la passerelle, les serveurs, les imprimantes avec IP fixe, etc.

Exclusions DHCP — Routeur d'Apt (Cisco IOS) Router# configure terminal ─── Exclusions VLAN 11 Administration ─── Router(config)# ip dhcp excluded-address 192.168.11.1 192.168.11.9 ← passerelle + adresses réservées ─── Exclusions VLAN 12 Personnel Accueil ─── Router(config)# ip dhcp excluded-address 192.168.12.1 192.168.12.9 ─── Exclusions VLAN 13 Clients WiFi ─── Router(config)# ip dhcp excluded-address 192.168.13.1 192.168.13.9 ← passerelle + AP WiFi Router(config)# end
📌 Syntaxe — ip dhcp excluded-address
ip dhcp excluded-address [première adresse] [dernière adresse]

Si une seule adresse est à exclure : ip dhcp excluded-address 192.168.11.1
Si une plage : ip dhcp excluded-address 192.168.11.1 192.168.11.9

4.3 Étape 2 — Créer les pools DHCP par VLAN

Un pool DHCP définit les paramètres attribués aux clients d'un VLAN donné. On crée un pool distinct pour chaque VLAN qui utilise DHCP.

Pool DHCP — VLAN 11 Administration Apt Router(config)# ip dhcp pool VLAN11-Admin-Apt ← nom du pool (sans espace) Router(dhcp-config)# network 192.168.11.0 255.255.255.240 ← réseau concerné + masque Router(dhcp-config)# default-router 192.168.11.1 ← passerelle distribuée aux clients Router(dhcp-config)# dns-server 192.168.15.2 ← serveur DNS interne (VLAN Serveurs) Router(dhcp-config)# lease 0 8 0 ← bail : 0 jours, 8 heures, 0 minutes Router(dhcp-config)# exit
Pool DHCP — VLAN 12 Personnel Accueil Apt Router(config)# ip dhcp pool VLAN12-Accueil-Apt Router(dhcp-config)# network 192.168.12.0 255.255.255.224 ← /27 = 255.255.255.224 Router(dhcp-config)# default-router 192.168.12.1 Router(dhcp-config)# dns-server 192.168.15.2 Router(dhcp-config)# lease 0 8 0 Router(dhcp-config)# exit
Pool DHCP — VLAN 13 Clients WiFi Apt Router(config)# ip dhcp pool VLAN13-WiFi-Apt Router(dhcp-config)# network 192.168.13.0 255.255.255.128 ← /25 = 255.255.255.128 Router(dhcp-config)# default-router 192.168.13.1 Router(dhcp-config)# dns-server 8.8.8.8 ← DNS public pour les invités WiFi Router(dhcp-config)# lease 0 2 0 ← bail court : 2h pour les clients de passage Router(dhcp-config)# exit
📌 Syntaxe de la commande lease
lease [jours] [heures] [minutes]

Exemples :
 • lease 0 8 0 → bail de 8 heures
 • lease 1 0 0 → bail de 1 jour
 • lease 7 → bail de 7 jours
 • lease infinite → bail infini (déconseillé en production)

5. Réservations DHCP — Attribuer une adresse fixe via DHCP

5.1 Principe de la réservation

Certains équipements ont besoin d'une adresse IP toujours identique (imprimante réseau, NAS, caméra IP gérée dynamiquement…), mais on veut quand même que cette adresse soit attribuée automatiquement via DHCP sans intervenir manuellement sur l'équipement.

La solution : la réservation DHCP. On associe l'adresse MAC (l'identifiant matériel unique) d'un équipement à une adresse IP fixe. Chaque fois que cet équipement demande une adresse, le serveur DHCP lui donne toujours la même.

🏨 Exemple Provence Évasion : L'imprimante réseau du VLAN 12 (Accueil Apt) doit toujours avoir l'adresse 192.168.12.5 pour que tous les postes puissent l'atteindre. Son adresse MAC est 0050.7966.6800. On crée une réservation DHCP pour garantir cette attribution.

5.2 Commandes de réservation sur Cisco IOS

Dans Packet Tracer et sur IOS, une réservation se configure comme un pool DHCP spécial, avec la commande host (à la place de network) et la commande hardware-address pour préciser l'adresse MAC.

Réservation DHCP — Imprimante VLAN 12 (Accueil Apt) Router(config)# ip dhcp pool RESERV-Imprimante-Accueil Router(dhcp-config)# host 192.168.12.5 255.255.255.224 ← adresse réservée + masque Router(dhcp-config)# hardware-address 0050.7966.6800 ← adresse MAC de l'imprimante Router(dhcp-config)# default-router 192.168.12.1 Router(dhcp-config)# dns-server 192.168.15.2 Router(dhcp-config)# exit
⚠ Important : L'adresse réservée (ici 192.168.12.5) doit aussi figurer dans les exclusions du pool général du VLAN 12, sinon le routeur pourrait l'attribuer à un autre client avant que l'imprimante ne se connecte. Vérifiez toujours la cohérence entre exclusions et réservations.

6. Vérification et dépannage DHCP

6.1 Commandes de vérification

Une fois la configuration DHCP en place, il faut vérifier qu'elle fonctionne correctement. Cisco IOS propose plusieurs commandes show très utiles.

Commandes de vérification DHCP — Routeur Apt ─── Voir tous les baux actifs (adresses actuellement attribuées) ─── Router# show ip dhcp binding IP address Client-ID / HW addr Lease expiration Type 192.168.12.10 0060.3E9A.1100 Jun 15 2026 10:00 AM Automatic 192.168.12.11 0060.3E9A.1101 Jun 15 2026 10:05 AM Automatic 192.168.12.5 0050.7966.6800 Infinite Manual 192.168.13.10 0060.5C4A.0001 Jun 15 2026 09:00 AM Automatic ─── Voir les statistiques DHCP (paquets échangés, erreurs) ─── Router# show ip dhcp statistics ─── Voir les pools DHCP configurés ─── Router# show ip dhcp pool ─── Voir les conflits d'adresses détectés ─── Router# show ip dhcp conflict ─── Supprimer tous les conflits enregistrés ─── Router# clear ip dhcp conflict *

6.2 Vérification côté client

Sur un PC connecté au réseau, on peut vérifier l'adresse IP obtenue via DHCP dans Packet Tracer depuis l'onglet Desktop → IP Configuration (mode DHCP sélectionné). On peut aussi utiliser la commande ipconfig dans l'invite de commandes du PC simulé.

Vérification côté client — Invite de commandes Packet Tracer C:\> ipconfig FastEthernet0 Connection:(default port) Connection-specific DNS Suffix..: Link-local IPv6 Address.........: FE80::201:97FF:FE5E:B401 IPv6 Address....................: :: IPv4 Address....................: 192.168.12.10 Subnet Mask.....................: 255.255.255.224 Default Gateway.................: 192.168.12.1 C:\> ipconfig /release ← libère le bail actuel (force un renouvellement) C:\> ipconfig /renew ← demande un nouveau bail (redéclenche DORA)

6.3 Problèmes courants et diagnostic

SymptômeCause probableAction corrective
Le client obtient 169.254.x.x (APIPA) Aucun serveur DHCP joignable Vérifier la configuration du pool, les VLANs et le trunk
Conflit d'adresse (ping échoue) Adresse non exclue attribuée deux fois show ip dhcp conflict puis clear ip dhcp conflict *
Pool DHCP vide (no addresses) Toutes les adresses sont épuisées ou trop d'exclusions Revoir les exclusions ou agrandir le pool (augmenter /masque)
Client DHCP d'un VLAN obtient l'adresse d'un autre VLAN Pool mal configuré (mauvais réseau déclaré) Vérifier la commande network de chaque pool
📌 APIPA — 169.254.x.x
Quand un client ne reçoit aucune réponse DHCP après plusieurs tentatives, il s'attribue automatiquement une adresse dans la plage 169.254.0.0/16 (APIPA — Automatic Private IP Addressing). C'est le signe que le serveur DHCP est inaccessible ou mal configuré.
