1. Pourquoi sécuriser un réseau segmenté ?

📍 Mise en situation : Provence Évasion dispose désormais de son réseau complet — 4 sites interconnectés, VLANs opérationnels, DHCP, DNS et partages réseau en service. Laurent Ferracci, PDG, soulève une inquiétude légitime : « Maintenant que tout est connecté, qui peut accéder à quoi ? Un client WiFi de l'hôtel de Roussillon peut-il atteindre les données comptables du siège d'Apt ? »
Votre rôle chez Axentis Informatique : définir et implémenter la politique de sécurité réseau via des listes de contrôle d'accès (ACL).

Les VLANs séparent logiquement les équipements sur le même switch. Mais dès qu'il y a du routage inter-VLAN ou inter-sites, tous les flux sont théoriquement possibles. Sans contrôle, un client WiFi peut potentiellement atteindre le serveur comptable. Les ACL (Access Control Lists) sont la réponse à ce problème : elles filtrent les paquets IP en entrée ou en sortie d'une interface de routeur.

📌 À retenir — Définition ACL
Une ACL est une liste ordonnée de règles (permit / deny) appliquée sur une interface de routeur. Chaque règle examine les paquets IP et décide de les autoriser ou de les bloquer. Le routeur parcourt les règles dans l'ordre et s'arrête à la première qui correspond (first match). Toute ACL Cisco se termine implicitement par un deny any (tout ce qui n'est pas autorisé explicitement est bloqué).

2. ACL standard vs ACL étendue

Cisco distingue deux grandes familles d'ACL :

🔵 ACL standard (1–99 ou 1300–1999)
  • Filtre uniquement sur l'adresse IP source
  • Syntaxe simple, peu de contrôle
  • À placer au plus près de la destination
  • Usage : bloquer un hôte ou un réseau entier
🟠 ACL étendue (100–199 ou 2000–2699)
  • Filtre sur IP source, IP destination, protocole, port
  • Beaucoup plus précise et flexible
  • À placer au plus près de la source
  • Usage : politique de sécurité fine (HTTP, DNS, Telnet…)
⚠️ Règle de placement : Une ACL standard ne connaît pas la destination → placez-la près de la destination pour éviter de bloquer des flux légitimes. Une ACL étendue connaît tout → placez-la près de la source pour économiser de la bande passante.

3. Le wildcard mask — masque inversé

Les ACL Cisco utilisent un wildcard mask (masque générique), différent du masque de sous-réseau. Il s'obtient en inversant bit à bit le masque habituel :

📐 Calcul du wildcard mask

Règle : wildcard = 255.255.255.255 − masque de sous-réseau

Réseau / hôteMasqueWildcardSignification
Un hôte précis255.255.255.2550.0.0.0Correspond exactement à cet hôte
192.168.10.0/24255.255.255.00.0.0.255Tous les hôtes du /24
10.0.0.0/8255.0.0.00.255.255.255Toutes les adresses 10.x.x.x
N'importe quelle adresse—255.255.255.255Équivaut au mot-clé any
Un hôte précis—0.0.0.0Équivaut au mot-clé host

4. Syntaxe des ACL Cisco IOS

4.1 ACL standard nommée

! Créer une ACL standard nommée Router(config)# ip access-list standard NOM-ACL Router(config-std-nacl)# permit 172.16.22.0 0.0.0.255 ! autoriser VLAN Direction Router(config-std-nacl)# deny 172.16.24.0 0.0.0.255 ! bloquer VLAN WiFi clients Router(config-std-nacl)# permit any ! autoriser le reste ! Appliquer sur une interface (sens IN = paquets entrant dans le routeur) Router(config)# interface GigabitEthernet0/0.24 Router(config-subif)# ip access-group NOM-ACL in

4.2 ACL étendue nommée

! Créer une ACL étendue nommée Router(config)# ip access-list extended NOM-ACL-EXT Router(config-ext-nacl)# deny ip 172.16.24.0 0.0.0.255 172.16.10.0 0.0.0.255 ! bloquer WiFi → serveurs Router(config-ext-nacl)# deny ip 172.16.23.0 0.0.0.255 any ! isoler caméras Router(config-ext-nacl)# permit ip any any ! autoriser tout le reste ! Appliquer sur l'interface source (sous-interface VLAN 24) Router(config)# interface GigabitEthernet0/0.24 Router(config-subif)# ip access-group NOM-ACL-EXT in

4.3 Vérification des ACL

Router# show access-lists ! liste toutes les ACL et compteurs Router# show ip interface GigabitEthernet0/0.24 ! ACL appliquée ? Router# show running-config | include access ! config ACL dans la running
📌 À retenir — Sens d'application (IN / OUT)
  • IN : le paquet est filtré avant d'entrer dans le routeur (il vient du réseau attaché à l'interface).
  • OUT : le paquet est filtré après que le routeur l'ait routé, avant qu'il ressorte par l'interface.
  • Dans 90 % des cas, on applique les ACL en IN sur l'interface de la source → économie de ressources CPU.

5. Ordre des règles et logique permit/deny

L'ordre des règles dans une ACL est critique. Le routeur teste chaque règle dans l'ordre, de la première à la dernière. Dès qu'une règle correspond, elle est appliquée et les règles suivantes sont ignorées.

#RègleActionCommentaire
1deny ip 172.16.24.0 0.0.0.255 172.16.10.0 0.0.0.255DENYWiFi → Serveurs : bloqué
2deny ip 172.16.23.0 0.0.0.255 anyDENYCaméras → partout : isolées
3permit ip any anyPERMITTout le reste : autorisé
—(implicit deny any any)DENYAjouté automatiquement
⚠️ Piège classique : si vous mettez permit ip any any en premier, toutes les règles deny suivantes ne seront jamais évaluées. Placez toujours les règles les plus spécifiques (deny précis) avant les règles générales (permit any any).

6. Stratégie de sécurité de Provence Évasion

📋 Cahier des charges sécurité — Laurent Ferracci :
  • Les clients WiFi (VLAN 24) ne doivent pas accéder aux serveurs internes (VLAN 10, siège Apt).
  • Les caméras IP (VLAN 23) doivent être totalement isolées — elles ne peuvent communiquer qu'avec le serveur de surveillance dédié.
  • La comptabilité (VLAN 22, direction) est accessible uniquement depuis le VLAN direction local et le siège.
  • Le personnel (VLAN 21) peut accéder aux services réseau normaux (fichiers, DNS, imprimantes).

6.2 Choix ACL standard ou étendue ?

Pour Provence Évasion, nous utiliserons principalement des ACL étendues nommées, car :

  • Les règles portent sur des paires source/destination précises (WiFi → Serveurs, Caméras → tout).
  • Certaines règles filtrent par protocole (TCP 22, TCP 445).
  • Les ACL étendues sont placées près de la source, ce qui optimise les performances du routeur.
