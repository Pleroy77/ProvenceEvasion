1. Pourquoi a-t-on besoin du DNS ?

📍 Mise en situation : L'infrastructure de Provence Évasion est maintenant interconnectée. Le siège d'Apt héberge un serveur NAS accessible à l'adresse 192.168.15.2, un NVR à l'adresse 192.168.15.3, et bientôt d'autres services. Laurent Ferracci doit retenir ces adresses pour accéder à ses serveurs. Ce n'est pas réaliste — et surtout, si une adresse change, tous les postes qui l'utilisent en dur devront être reconfigurés. La solution : un serveur DNS, installé à Apt, qui permet d'accéder à nas.provence-evasion.local au lieu de mémoriser 192.168.15.2.

1.1 Le problème des adresses IP

Une adresse IP comme 192.168.15.2 est parfaitement compréhensible pour un équipement réseau, mais elle est difficile à retenir pour un être humain. De plus :

  • Une adresse IP peut changer (renumérotage du réseau, remplacement d'un serveur).
  • Un même service peut être réparti sur plusieurs serveurs avec des adresses différentes.
  • Certaines applications et protocoles nécessitent de connaître un nom plutôt qu'une adresse (ex : envoi d'e-mail, certificats SSL, etc.).

Il fallait donc inventer un système permettant d'associer un nom lisible (appelé nom de domaine) à une adresse IP. Ce système, c'est le DNS — Domain Name System.

📌 À retenir — Définition du DNS

Le DNS (Domain Name System) est un service réseau qui traduit des noms de domaine en adresses IP, et inversement. C'est l'annuaire téléphonique d'Internet : vous cherchez un nom, il vous donne le numéro (l'adresse IP).

Exemple : quand vous tapez www.google.com dans votre navigateur, le DNS traduit ce nom en une adresse IP (par exemple 142.250.185.68) que votre machine utilise pour établir la connexion.

1.2 Un peu d'histoire — avant le DNS

Avant 1983, les ordinateurs connectés à ARPANET (l'ancêtre d'Internet) utilisaient un simple fichier texte appelé HOSTS.TXT, maintenu manuellement par le Stanford Research Institute. Ce fichier associait chaque nom de machine à son adresse IP.

Avec la croissance d'Internet, cette approche est devenue ingérable : le fichier était mis à jour plusieurs fois par semaine, chaque machine devait le télécharger régulièrement, et les conflits de noms se multipliaient. Le DNS, inventé en 1983 par Paul Mockapetris, a résolu ce problème en créant un système hiérarchique et distribué.

📁 Le fichier hosts existe encore ! Sur chaque machine (Windows, Linux, macOS), le fichier /etc/hosts (ou C:\Windows\System32\drivers\etc\hosts) est consulté avant le DNS. C'est un mécanisme de résolution locale prioritaire. On peut y forcer des associations nom/IP — utile pour les tests, mais dangereux si un logiciel malveillant le modifie.

2. Architecture du DNS — noms de domaine et hiérarchie

2.1 La structure hiérarchique des noms de domaine

Un nom de domaine complet est appelé FQDN (Fully Qualified Domain Name). Il se lit de droite à gauche, du plus général au plus précis :

Décomposition d'un FQDN
nas.provence-evasion.local. │ │ │ └── Point final (racine, souvent implicite) │ │ └─────── TLD : domaine de premier niveau (.local, .com, .fr…) │ └───────────────────────── Domaine de second niveau (provence-evasion) └───────────────────────────── Sous-domaine / nom d'hôte (nas) Autre exemple : www.google.com. │ │ │ │ │ └── TLD : .com (domaine commercial) │ └────────── Domaine : google └────────────── Hôte : www

2.2 La hiérarchie DNS — des serveurs racines aux serveurs locaux

Le DNS est organisé comme un arbre inversé. À sa racine se trouvent les serveurs racines (root servers), au nombre de 13 groupes dans le monde, qui connaissent les adresses des serveurs responsables de chaque TLD (.com, .fr, .net…). Ensuite, pour chaque domaine, un serveur faisant autorité (authoritative server) possède les enregistrements définitifs.

Dans la pratique, pour ne pas interroger la racine à chaque demande, les serveurs DNS utilisent un cache : ils mémorisent les réponses pendant une durée définie par le TTL (Time To Live). Une fois le TTL expiré, l'information est considérée périmée et une nouvelle requête est effectuée.

📌 À retenir — Résolution récursive vs itérative

Quand un client demande à son serveur DNS la résolution d'un nom, deux modes existent :

  • Résolution récursive : le serveur DNS prend en charge toute la résolution pour le client — il interroge les autres serveurs DNS jusqu'à obtenir la réponse finale, puis la retourne au client. C'est le mode habituel pour les clients.
  • Résolution itérative : le serveur répond "je ne sais pas, mais demande à ce serveur-là". Le client (ou un forwarder) enchaîne les requêtes lui-même. C'est le mode utilisé entre serveurs DNS de la hiérarchie.

Dans le projet Provence Évasion, les postes clients interrogent le serveur DNS interne (Apt) en mode récursif. Si le serveur d'Apt ne connaît pas la réponse (ex : une adresse externe comme www.google.com), il la transfère (forwarder) vers un résolveur public (ex : 8.8.8.8).

3. Zones DNS — résolution directe et inverse

3.1 La zone de résolution directe

✅ Zone directe (Forward Lookup Zone)

La zone de résolution directe répond à la question : "Quelle est l'adresse IP de ce nom ?"

C'est la zone la plus utilisée. Elle contient des enregistrements qui associent des noms de domaine à des adresses IP (enregistrements de type A principalement).

Exemple : nas.provence-evasion.local → 192.168.15.2

3.2 La zone de résolution inverse

🔁 Zone inverse (Reverse Lookup Zone)

La zone de résolution inverse répond à la question : "Quel est le nom associé à cette adresse IP ?"

C'est l'opération symétrique de la résolution directe. Elle utilise des enregistrements de type PTR (Pointer Record). Le nom de la zone inverse est construit en inversant les octets du réseau, suivi de .in-addr.arpa.

Exemple : Pour le réseau 192.168.15.0/24, la zone inverse est 15.168.192.in-addr.arpa. L'enregistrement PTR associe 2 (le dernier octet de 192.168.15.2) à nas.provence-evasion.local.

📌 À retenir — Pourquoi la zone inverse ?

La zone inverse est utilisée par :

  • Les outils de diagnostic réseau (nslookup, dig, ping -a) qui affichent des noms au lieu d'adresses IP.
  • Certains services de messagerie qui vérifient que l'IP d'un serveur correspond bien à un nom (anti-spam).
  • Les journaux (logs) système qui deviennent bien plus lisibles avec des noms que des adresses.

Dans l'atelier Packet Tracer (S7), vous configurerez la zone inverse pour le réseau des serveurs d'Apt : 15.168.192.in-addr.arpa.

4. Mécanisme de résolution DNS — étape par étape

4.1 Le processus de résolution

Voyons en détail ce qui se passe lorsqu'un poste du personnel de Roussillon tente d'accéder au NAS d'Apt en tapant \\nas.provence-evasion.local dans son explorateur Windows :

1
Cache local : Windows vérifie d'abord son propre cache DNS (ipconfig /displaydns). Si le nom a déjà été résolu récemment et que le TTL n'est pas expiré, la réponse est utilisée immédiatement sans aucune requête réseau.
2
Fichier hosts : Windows consulte le fichier C:\Windows\System32\drivers\etc\hosts. Si une entrée manuelle correspond, elle est utilisée (prioritaire sur le DNS).
3
Requête DNS vers le résolveur : Le poste envoie une requête DNS (UDP, port 53) au serveur DNS configuré dans ses paramètres IP — ici, 192.168.15.1 (serveur DNS d'Apt, obtenu via DHCP).
4
Le serveur DNS d'Apt répond : Il est autorité pour la zone provence-evasion.local. Il retrouve l'enregistrement A de nas dans sa zone directe et renvoie 192.168.15.2.
5
Mise en cache et connexion : Le poste mémorise la réponse dans son cache pour la durée du TTL. Il utilise ensuite l'adresse 192.168.15.2 pour se connecter au NAS via le routage inter-sites.

4.2 Et si le nom n'existe pas en interne ?

Que se passe-t-il si un employé de Provence Évasion tape www.meteo.fr dans son navigateur ? Le serveur DNS interne d'Apt ne connaît pas ce nom (il n'appartient pas à la zone provence-evasion.local). Dans ce cas :

Résolution d'un nom externe — avec forwarder
PC-Roussillon ──→ [Serveur DNS Apt : 192.168.15.1] │ │ "Je ne connais pas www.meteo.fr" ↓ [Forwarder : 8.8.8.8 (Google DNS)] │ │ Résolution sur Internet (hiérarchie DNS) ↓ Réponse : www.meteo.fr → 217.160.x.x │ ↓ PC-Roussillon ←── Réponse : 217.160.x.x

Le serveur DNS d'Apt est configuré avec un forwarder (redirecteur) — une adresse de serveur DNS externe (ex : 8.8.8.8 de Google, ou 1.1.1.1 de Cloudflare) vers lequel il transmet les requêtes qu'il ne peut pas résoudre lui-même.

📌 À retenir — Le port 53

Le DNS utilise le port 53, en UDP pour les requêtes classiques (réponses < 512 octets) et en TCP pour les transferts de zone entre serveurs DNS ou les réponses volumineuses. En termes de sécurité réseau, les ACL de Provence Évasion devront autoriser le trafic UDP/TCP port 53 entre les postes clients et le serveur DNS d'Apt.
